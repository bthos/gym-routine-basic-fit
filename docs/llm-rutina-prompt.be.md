# Стварэнне Руціны з дапамогай LLM

Гэта кіраўніцтва ператварае праграму трэніровак, якая існуе толькі ў вашай галаве
(ці ў нататках трэнера), у файл `rutina.json` — фармат даных, які вызначае схема
руціны праекта. Для гэтага трэба запоўніць кароткі спіс патрабаванняў, уставіць
сабраны prompt у **любы** LLM-чат (ChatGPT, Claude, Gemini — усё, што ёсць пад рукой,
на любой прыладзе), і праверыць адказ з дапамогай лакальнага скрыпту.

Веданне JSON не патрабуецца. Без уліковага запісу, без ключа API — гэта працуе ў
звычайным акне чата браузера.

- **Схема (крыніца ісціны):** `data/schema/rutina.schema.json`
- **Валідатар:** `scripts/validate-rutina.js`, запуск праз `npm run validate-rutina -- <шлях>`
- **Даведачны прыклад:** `data/examples/` (рэальны `rutina.json`, які праходзіць валідатар)
- **Каталог абсталявання, на які спасылаецца схема:** `data/equipment.json`

> Блок SCHEMA, уключаны ў гэта кіраўніцтва (гл. [Крок 3](#крок-3--скапіруйце-поўны-шаблон-prompt)),
> з'яўляецца *анатаваным чарнавіком* у тым жа фармаце, які ўстанаўлівае
> `data/schema/rutina.schema.json`, зручным для чытання чалавекам/LLM. Калі яны
> калі-небудзь не супадаюць, перамагае файл схемы — гэта кіраўніцтва з'яўляецца
> дакументацыяй, а не валідатарам.

## Як гэта працуе

```
1. Запоўніце спіс патрабаванняў (звычайны тэкст, ніжэй)
2. Прыкладзіце спіс абсталявання (поўны каталог ці падмноства вашага залы)
3. Скапіруйце поўны шаблон prompt, устаўце свой спіс патрабаванняў у REQUEST
4. Устаўце ўсё ў любы LLM-чат
5. Скапіруйце адказ JSON ад LLM у файл
6. Праверце:  npm run validate-rutina -- шлях/да/вашай-rutina.json
      ✓ прайшло  → гатова, файл гатовы да імпарту
      ✗ не прайшло → устаўце тэкст памылкі назад у LLM-чат наступным паведамленнем,
                     папрасіце выправіць і паўторна вывесці поўны JSON, вярніцеся да кроку 5
7. Імпартуйце ў PWA (калі яна з'явіцца) — ці захоўвайце файл як ёсць; праверaны
   rutina.json з'яўляецца поўным і карысным артэфактам сам па сабе
```

Цыкл выпраўлення ў кроках 5–6 — гэта рэальны механізм надзейнасці. Мы не спрабуем
атрымаць дасканалы JSON-аб'ект ад LLM з першага спробу — мы робім збой *зразумелым
і самакарэктным*. Паведамленні аб памылках валідатара напісаны так, каб вы маглі
ўставіць іх прама ў чат.

## Крок 1 — Запоўніце спіс патрабаванняў

Скапіруйце гэты блок, запоўніце кожную радок (звычайны тэкст, без JSON) і захоўвайце
яго так, каб можна было ўставіць у секцыю `REQUEST` Кроку 3.

```text
1. Для каго гэта / назва праграмы:
2. Асноўная мэта ў гэтай фазе:
3. Дзён у тыдзень:
4. Бюджэт часу сесіі:
5. Траўмы / рухі, якіх трэба пазбягаць (напішыце "няма" калі няма):
6. Мэтавы трэнажорны зал (назва ці id — гл. data/gyms.json):
7. Мова для выходнага тэксту:
8. Экспарт папярэдняга прагрэсу (неабавязкова — устаўце Markdown з экрана экспарту PWA або пакіньце пустым):
```

| # | Поле | Прыклад | Абавязковае |
|---|------|---------|-------------|
| 1 | Для каго гэта / назва праграмы | "Іван — Фаза 2" | так |
| 2 | Асноўная мэта ў гэтай фазе | "Гіпертрафія, больш аб'ёму" | так |
| 3 | Дзён у тыдзень | 4 | так |
| 4 | Бюджэт часу сесіі | "45-60 хвілін" | так |
| 5 | Траўмы / рухі, якіх трэба пазбягаць | "Пазбягаць ваеннага жыму праз правае плячо" | не — напішыце "няма" калі няма |
| 6 | Мэтавы трэнажорны зал | "Bulevar Louis Pasteur 20, Малага" | так — вызначае, якое падмноства абсталявання прымяняецца |
| 7 | Мова для выходнага тэксту | "Беларуская" | так |
| 8 | Экспарт папярэдняга прагрэсу | Markdown з экрана экспарту PWA | не — пакіньце пустым пакуль PWA не існуе |

## Крок 2 — Прыкладзіце спіс абсталявання

Поле 6 (мэтавы трэнажорны зал) вызначае, якія ID абсталявання сапраўды можна
выкарыстоўваць — у трэнажорным зале ёсць толькі тыя тренажоры, якія ёсць. Выберыце
**адзін** з двух варыянтаў:

**Варыянт А — поўны каталог, дазвольце LLM самастойна адфільтраваць (прасцейшы).**
Прыкладзіце ўвесь файл `data/equipment.json` (ці ўстаўце яго змесціва) і паведаміце
LLM у `REQUEST`, які ID трэнажорнага залы вы выкарыстоўваеце. Кожны запіс абсталявання
ўтрымлівае масіў `gyms` (напр. `"gyms": [1, 2, 3, 4, 5, 6, 7]`); шаблон prompt у
Кроку 3 ужо ўказвае LLM выбіраць толькі абсталяванне, у масіве `gyms` якога ёсць ваш
ID трэнажорнага залы.

**Варыянт Б — адфільтраванае падмноства (менш токенаў, карысна для LLM з малым кантэкстам).**
Адкрыйце `equipment-catalog.html` у браўзеры (`npm run serve`, затым перайдзіце на
`http://localhost:3000/equipment-catalog.html`), знайдзіце значок вашага трэнажорнага
залы на кожнай картцы абсталявання і скапіруйце `id`/`modelCode` і назву кожнага
тренажора, які мае значок вашага залы, у кароткі спіс. Устаўце гэты спіс у
`AVAILABLE EQUIPMENT` замест поўнага файла.

Абодва варыянты падыходзяць — Варыянт А патрабуе менш ручной працы, Варыянт Б
стварае больш кароткі prompt.

## Крок 3 — Скапіруйце поўны шаблон prompt

Гэта ўвесь шаблон, гатовы для ўстаўкі ў LLM-чат. Запоўніце два маркеры ў дужках
(`<<< ... >>>`) перад адпраўкай: спіс абсталявання з Кроку 2 і запоўнены спіс
патрабаванняў з Кроку 1.

````text
### ROLE
You are a strength & conditioning coach. You will output ONLY a single JSON
object — no prose, no markdown fences, no explanation — conforming exactly
to the schema below. If a requested field has no natural value, omit it
only if the schema marks it optional (shown with `?`); never invent
equipment ids that are not in the AVAILABLE EQUIPMENT list.

### SCHEMA
{
  "schemaVersion": 1,
  "program": {
    "name": "string",              // e.g. "Nombre — Fase 2"
    "phaseName": "string",
    "phaseNumber": "number",
    "gymId": "number",             // joins data/gyms.json
    "durationWeeks": "number",
    "lastUpdated": "YYYY-MM-DD"
  },
  "phaseInfo": {
    "objective": "string",
    "intensityPercent": "string",  // e.g. "70-75%"
    "restSeconds": "string",       // e.g. "45-60"
    "frequencyPerWeek": "number"
  },
  "warmup":   { "durationMinutes": "string", "steps": ["string", ...] },
  "cooldown": { "durationMinutes": "string", "steps": ["string", ...] },
  "days": [
    {
      "label": "string",           // e.g. "Lunes", "Full Body A"
      "intro": "string?",
      "exercises": [
        {
          "name": "string",
          "muscleGroups": ["string", ...],
          "equipmentId": "string", // MUST exist in AVAILABLE EQUIPMENT below
          "sets": "number",
          "reps": "string",        // e.g. "12-15" — a string, not a number (ranges)
          "restSeconds": "number",
          "intensity": "string?",  // e.g. "RIR 1-2"
          "technique": ["string", ...],  // optional
          "videoQuery": "string?"  // search query OR full URL
        }
      ]
    }
  ],
  "rules": ["string", ...],
  "notes": [ { "title": "string", "body": "string" } ],
  "progression": [                 // optional — omit entirely if not applicable
    {
      "month": "string",
      "weeks": [ { "label": "string", "focus": "string", "deload": "boolean?" } ]
    }
  ]
}

Do NOT include a "summaryTable" field — it does not exist in the schema and
will fail validation. Any summary view is derived from "days" elsewhere.
`?` on a field name (or a "// optional" comment) means optional; every other field is required.

### AVAILABLE EQUIPMENT (use only these ids — do not invent equipment)
<<< устаўце сюды свой спіс абсталявання з Кроку 2 >>>

### REQUEST
<<< устаўце сюды свой запоўнены спіс патрабаванняў з Кроку 1, літаральна >>>

### OUTPUT
Return the JSON object now. Nothing else.
````

Двойнае ўключэнне "вывадзіць толькі JSON, нічога іншага" у пачатку (`ROLE`) і ў канцы
(`OUTPUT`) — гэта наўмысна. Гэта адзіная найбольш эфектыўная інструкцыя для атрымання
прасавальнага выніку ад чат-LLM, які інакш захоча дадаць сяброўскі ўступ ці загарнуць
адказ у блок markdown.

## Крок 4 — Праверце атрыманы адказ

Захавайце адказ LLM як файл `.json`, затым запусціце:

```bash
npm run validate-rutina -- шлях/да/вашай-rutina.json
```

Што вы ўбачыце:

| Вынік | Прыклад выніку | Што рабіць |
|---|---|---|
| Поспех | `✓ Valid rutina: 4 days, 22 exercises` | Гатова — файл падрыхтаваны |
| Памылка схемы | `days[1].label: required` | Адна радок на парушэнне, шлях у стылі JSON-pointer. Устаўце ўвесь блок памылак назад у LLM-чат і папрасіце выправіць і паўторна вывесці поўны JSON. |
| Памылка ID абсталявання | `days[2].exercises[0].equipmentId "g3-xx" not found in data/equipment.json` | Тое ж — устаўце назад літаральна. |
| Шлях не ўказаны | Радок выкарыстання + прыклад выкліку | Паўторна запусціце з аргументам шляху. |

Тэкст памылкі наўмысна напісаны так, каб яго можна было ўставіць — вам не трэба
перакладаць ці рэзюмаваць яго для LLM, проста скапіруйце ўвесь блок як наступнае
паведамленне ў чаце.

## Крок 5 — Імпарт у прыкладанне

Як толькі PWA (`rutina-pwa`) з'явіцца, вы зможаце альбо:

- Уставіць праверaны JSON непасрэдна ў экран "Імпарт руціны", альбо
- Загрузіць файл `.json` праз імпарт файлаў PWA.

Да таго часу праверaны `rutina.json` ужо з'яўляецца поўным самастойным артэфактам —
вы можаце яго чытаць, параўноўваць з папярэдняй версіяй ці перадаць камусьці іншаму.
Нічога ў ім не залежыць ад існавання PWA.

## Вырашэнне праблем

- **LLM дадаў сяброўскі ўступ ці загарнуў JSON у блок markdown.** Адпраўце зноўку
  з напамінам: "Output ONLY the JSON object, no markdown fence, no explanation."
  Гэта рэдкасць, калі вы захавалі секцыі `ROLE` і `OUTPUT` з Кроку 3 цэлымі — гэтае
  дубліраванне існуе менавіта для прадухілення гэтага.
- **Адказ быў абарваны пасярэдзіне JSON (вельмі доўгія праграмы).** Папрасіце LLM
  "працягнуць з таго месца, дзе спыніліся, усё яшчэ выводзячы толькі JSON" — большасць
  чат-інтэрфейсаў аднавіцца з месца абрыву.
- **ID абсталявання не знойдзены.** Праверце, ці паходзіць ID са спісу Кроку 2 для
  вашага рэальнага трэнажорнага залы, а не з іншага залы ці прыдуманага/перайменаванага
  ID. ID абсталявання ніколі не з'яўляюцца свабодным тэкстам — яны павінны дакладна
  адпавядаць `data/equipment.json`.
- **Не ўпэўнены ў ID свайго трэнажорнага залы?** Праверце `data/gyms.json` — кожны
  трэнажорны зал мае лічбавы `id` і зразумелыя назву/адрас (`name`/`address`).

## Абнаўленне праграмы

Калі праграма загружана, вы можаце замяніць яе ці выдаліць з укладкі **Праграма**:

1. Адкрыйце ўкладку **Праграма** і пракруціце да канца агляду.
2. Націсніце **Reemplazar programa** (замяніць праграму), каб перайсці на экран імпарту і ўставіць новы rutina.json.
3. Ці націсніце **Eliminar programa** (выдаліць праграму), каб ачысціць актыўную праграму і вярнуцца да пачатковага стану.

Гісторыя сесій захоўваецца пры выдаленні — запісы застаюцца, але больш не будуць звязаны з праграмай.
